## P1 风险

### P1-1. 失败的 Dreamina 任务卡片仍露出 `id:` + 截断 submit_id (用户可见工程词)

`web/factory-task.jsx:617-620`:

```jsx
{failed && (task.kind || "").startsWith("dreamina.") && task.payload?.submit_id && (
  <div ...>
    <span style={{ fontSize: 10.5, color: T.muted2, fontFamily: "monospace" }}>
      id: {String(task.payload.submit_id).slice(0, 12)}…
    </span>
```

T-062 的扫描词单里有 `submit_id`, 但 QA 是空状态截图 (没有真出过失败的 Dreamina 任务), 所以扫不到这条 conditional 露出. 结果是: 一旦"重新生成数字人/即梦"失败留底, 全局任务 dock 上会出现 `id: 88a7c9b1c4f2…` 这种纯工程串号. 老板要拍视频/录设置页时只要踩到一次失败任务, 录到的画面里就有 12 位 hash. 这是 T-062 二轮清理本来想消的体感.

QA 同时跑过的可见词扫描里也没扫 `id: ` 前缀 (只扫 `submit_id` 字面量), 因此这条会一直安全溜过. 影响面: 数字人/即梦的失败长尾 (>0 秒就会出现).

建议改成 `任务编号 ··` + 4 位短码, 或者直接去掉这一行 -- 反正它下面一行就是 "🔍 重查即梦" 按钮, 失败编号对老板没用.

### P2-2. `/api/ai/health` 默认行为没有显式回归测试

D-126 只新增了 "短探活时, calls == [{timeout: 3.0, llm_max_retries: 0}]" 的断言 (`tests/test_health_api.py:46`) 和 "短探活 kwargs 透传到 ClaudeOpusClient" 的隔离测试 (`tests/test_ai_routing.py:84-101`). 默认 `get_ai_info()` 不传任何参数时, `timeout=None` / `llm_max_retries=None` 是否仍然回到老的 120s + 1 次重试 -- 没有红绿测试守住. 一旦后续有人把 `get_ai_info` 默认值改成 `timeout: float = 3.0`, 设置页"重探"按钮就会变成短探, 没人察觉.

实测我自己跑了一次 `/api/ai/health?fresh=1` -> 6.48s 真打成功, 行为对. 但运行时探到 != 测试断言.

建议补一条:

```python
def test_get_ai_info_default_uses_full_probe_options(monkeypatch):
    captured = {}
    class FakeClaudeClient:
        def __init__(self, **kwargs):
            captured.update(kwargs)
        def health(self):
            return {"ok": True, "reply": "好"}
    monkeypatch.setattr("backend.services.settings.get_all", lambda: {"ai_engine": "opus"})
    monkeypatch.setattr(ai_mod, "ClaudeOpusClient", FakeClaudeClient)
    ai_mod.get_ai_info()
    assert captured.get("timeout") is None
    assert "llm_max_retries" not in captured  # 默认走 client class 默认 = 1
```

不烧 credits, 一行守门测试.

### P2-3. `今日用量` 的"K字"换算阈值导致低用量永远显示 `0K字`

`web/factory-write.jsx:69`:

```jsx
<StatBlock label="⚡ 今日用量" value={Math.round(todayTokens * 0.7 / 1000) + "K字"} sub={`今日 · 约 ¥${todayCost.toFixed(2)}`} />
```

`Math.round(todayTokens * 0.7 / 1000)` 在 `todayTokens` < 1429 时四舍五入回 0, 显示 `0K字`. 实际生成 1 条投流文案大约消耗 800-1500 tokens, 也就是说前两条文案产出后, 这格仍然会显示 `0K字 · 今日 · 约 ¥0.0X`. 跟"📝 今日产出"里"2 条文案"对不上, 看着像"统计没接上".

实测截图 `/tmp/_ui_shots/t062_fix_write.png` 第一行就是 `0K字 / ¥0.00`, 当时 todayTokens=0 没毛病; 但量级一上来就会暴露. 这是 D-069 把 token 转字数时的边界 UX, 不是 T-062 引入的, 但 T-062 改文案时把这个口径正式定下来了 -- 留作残余风险.

短期建议: < 1000 字时改用 "字" 直显 (例如 `Math.round(todayTokens * 0.7) + "字"`), > 1000 字再切 K.

或者直接换一个不需要单位转换的口径, 比如 "今日生成 X 条" -- 这跟 "📝 今日产出" 重复, 不合适. 还是建议保留字数, 但解决低量级回 0 的问题.
