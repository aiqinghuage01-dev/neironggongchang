## 一、为什么再改一次 (业务 WHY)

D-063 把"爆款改写"接进做视频页后, 改写按钮变成 popover 三选一:

```
[🎬 做数字人] [✍️ 改写优化 ▾]
                    ├─ 🎙️ 录音改写 (推荐)
                    ├─ ✍️ 爆款改写
                    └─ 🔥 热点改写
```

清华哥指出: **这个三选一本身是错的**, 因为 3 个 skill 的输入语义不一样:

| Skill | API 字段 | 真正的内容来源 | 语义动作 |
|---|---|---|---|
| ✍️ 爆款改写 | `text` ("原爆款") | **别人发的爆款** | 抓 DNA → 换皮保结构 |
| 🎙️ 录音改写 | `transcript` ("录音转写") | **自己录的音** | 删口头禅、规整、不改观点 |
| 🔥 热点改写 | `hotspot` ("热点话题") | **今日热点新闻** | 拆多角度 → 出口播 |

popover 把"用户从一段提取出的文案出发"分到 3 条路, 但:
- 提取出来的是别人的爆款 → 走录音改写? **错**, 这不是录音
- 提取出来的是某条具体文案 → 走热点改写? **错**, 这不是热点新闻
- 提取出来的就是别人爆款 → 走爆款改写 ✅ **唯一对的**

**根因**: v1 把 3 个改写 skill 当成"3 种改写动作"摆平级, 但其实它们是 **3 个不同的内容起点**.

清华哥原话: "提取文案这件事情, 本质上应该就是一个爆款改写。提取的文案它不存在是录音, 也不存在是热点。"

## 二、用户故事 (4 条主要路径)

| 故事 | 入口 | 对应 skill |
|---|---|---|
| **A. 我刷到一条爆款想抄** | 贴抖音/小红书链接 | 爆款改写 |
| **B. 我自己录了一段, 讲得有点散** | 贴录音转写 | 录音改写 |
| **C. 我看到今天有个热点想拍** | 选热点卡 | 热点改写 |
| **D. 我已经写好了草稿** | 贴文案 | (可选)爆款改写洗 / 直接做 |

每条路径走完后都汇到 **Step 2: 数字人 + 配音**, 然后 Step 3 渲染.

**反例 (排除掉的伪需求)**:
- ❌ "我从别人视频提取了文案, 想用录音改写整理一下" — 不存在这种语义
- ❌ "我从别人视频提取了文案, 想当成热点改写一遍" — 不存在这种语义
- ❌ "我录了一段, 想用爆款改写公式洗一下" — 录音改写已经包含规整, 不需要套别人爆款的 DNA

## 三、方案 D — 按"内容来源" tab 分流

### 3.1 整体布局

```
                    🎬 做视频 · 第 1 步: 给我点东西

         你要做的视频从哪来? 选一个起点 ↓

    ┌────────────────────────────────────────────────────────┐
    │  [📹 别人的视频]  [🎙️ 我自己录的]  [🔥 今天的热点]  [✏️ 已写好的文案]  │
    │      默认                                                  │
    └────────────────────────────────────────────────────────┘

    (下面 tab 内容根据选中切换 — 详见 §4)

    ──────────────────────────────────────────────────────────

    所有 tab 走完都汇到 Step 2: 选数字人 + 配音
```

**关键点**:
- Tab 是 segmented (像 iOS 那种), 不是 popover, 不是大卡片网格
- 默认选中"别人的视频"(用户最高频场景)
- 4 个 tab 是**互斥的内容起点**, 不是"改写方式"

### 3.2 各 tab 内容详解

#### Tab 1: 📹 别人的视频 (默认, 对应爆款改写场景)

```
你刷到的那条爆款, 贴链接/转发文给我

┌─────────────────────────────────────────────────────────┐
│  [textarea 大圆角]                                         │
│   粘抖音/小红书/B 站链接, 或者抖音转发文 (我自动找链接)         │
│                                                          │
└─────────────────────────────────────────────────────────┘
状态: 🔗 看起来是短视频链接 · 来源: 抖音

[📎 提取并洗成我的爆款]     [📎 只提取不洗]
   主按钮 (brand)             次按钮 (outline)

提示: "提取走轻抖 ASR, 1-3 分钟"
```

**3 种 textarea 状态**(沿用现有 `URL_PATTERN` 识别):

| 状态 | 主按钮 | 次按钮 |
|---|---|---|
| 空 | `↑ 先粘链接` (灰 disabled) | – |
| 纯 URL | `📎 提取并洗成我的爆款 →` | `📎 只提取不洗 (跳改写)` |
| 混合 (转发文) | `📎 提取并洗成我的爆款 →` (从链接) | `📎 只提取不洗` |
| 误粘文案 (没 URL) | (灰提示: "这里只接链接, 文案请切到 ✏️ tab") | – |

**主按钮行为**:
1. ASR 提交 + 60 × 5s 轮询
2. 拿到 text → `localStorage.setItem("baokuan_seed_text", text)`
3. `onNav("baokuan")` → 自动跳爆款改写页 input step (textarea 自动填好)
4. 用户在爆款改写选模式 + 出 V1/V2 → 选版本 → "做成视频" CTA 带回 Step 2

**次按钮行为**:
1. ASR 同上
2. 拿到 text → `setScript(text)` 留在 Step 1, 但**自动切到 ✏️ tab** (因为现在变成"已有文案"模式)
3. 用户决定: 直接做数字人 / 还是去爆款改写

#### Tab 2: 🎙️ 我自己录的 (对应录音改写场景)

```
你录的那段音, 贴转写文本给我 (录音文件上传暂不支持, 用其他工具先转)

┌─────────────────────────────────────────────────────────┐
│  [textarea]                                              │
│   把录音转写后的文字贴这里, 我帮你删口头禅、加结构、不改观点   │
│                                                          │
└─────────────────────────────────────────────────────────┘
状态: 380 字 · 现场录音感 · 适合 90 秒口播

[✏️ 帮我整理这段口播 →]
   主按钮 (brand) → 跳录音改写
```

**为什么不接录音文件上传?**
- 当前 backend 的轻抖 ASR 只接受短视频链接, 不接独立音频文件
- 用户场景: 自己录了 → 通常已经在飞书妙记/讯飞听见/手机自带转写里得到文本了
- 真要做"录音文件上传 → 转写"是单独的 backlog (需要新接 ASR 服务), 不在这次优化范围

**如果用户想直接录音?** 给一个软引导:
```
还没转写? → 推荐去 飞书妙记 / 讯飞听见 转一下再贴回来
```
(链接形式, 不嵌入)

**主按钮行为**:
- `localStorage.setItem("voicerewrite_seed_transcript", text)`
- `onNav("voicerewrite")` → 自动跳录音改写页 input step

#### Tab 3: 🔥 今天的热点 (对应热点改写场景)

```
小华今天给你选的 3 个值得拍的:

┌─────────────────────────────────────────────────────────┐
│ [1] AI 客服集体下岗?一线从业者发声      匹配度 98%       │
│     ┃ 匹配原因: 你擅长 AI 落地 + 个人视角                  │
│     ┃ 建议渠道: 🎵 抖音  📺 视频号                       │
│     ┃                          [只塞文案] [📸 拍这条 →]   │
│ [2] ...                                                  │
│ [3] ...                                                  │
└─────────────────────────────────────────────────────────┘

不喜欢? 全部 (6) →    或者 🌙 启用夜班抓更多

或: 自己粘一条热点新闻
┌─────────────────────────────────────────────────────────┐
│  [textarea]  贴新闻链接 / 标题 / 概要                    │
└─────────────────────────────────────────────────────────┘
[🔥 改成我的角度 →]
```

完全复用现有 `HotPickCard` + `NightHotFlywheel` (D-062nn-C2/C3 已做好), 只是位置从原来的 textarea 下方挪到 tab 3 内部.

#### Tab 4: ✏️ 已写好的文案

```
你自己写好的稿, 贴这里

┌─────────────────────────────────────────────────────────┐
│  [textarea]                                              │
│   把已经写好的口播文案贴这里                                │
│                                                          │
└─────────────────────────────────────────────────────────┘
状态: 420 字 · ~120 秒口播

[🎬 做数字人 →]    [✍️ 爆款改写洗一下]
   主按钮              次按钮 (留口子给"想再优化的"用户)
```

**为什么 tab 4 留爆款改写次按钮?**
- 用户写完草稿可能想"对照爆款公式洗一遍" — 这是合理的爆款改写场景 (用户自己写的就是"原爆款", 让 skill 重新组织)
- 但默认 CTA 是直接做, 因为"写好了"语义上就是想做了

## 五、边界 case 处理

### 5.1 用户在 tab 1 提取后想改 tab?
- 提取出的文案进入 textarea, 自动切到 tab 4 (因为现在已经是"已有文案"状态)
- 顶部加 banner: "✓ 从 [URL] 提取了 N 字 · 已切到 ✏️ tab, 你可以直接做或先洗"

### 5.2 用户切 tab 时已经填了内容?
- 切之前给 confirm: "切走会清空当前输入, 确定吗?" (轻量, 不要硬拦)
- 如果已经在 ASR 中 (extracting): 切 tab 暂停轮询 + 提示"提取已取消"

### 5.3 tab 3 没有热点?
- `hotTopics === null` (loading): 显占位 skeleton
- `hotTopics.length === 0`: 显 NightHotFlywheel (启用夜班/立即抓), 同时显"或者自己粘一条新闻"
- `0 < length < 3`: 显已有的 + 占位卡 (现有逻辑不动)
- `length ≥ 3`: 显 top 3 + "全部 (N) →"

### 5.4 已经在 D-063 接入的爆款改写, 怎么承接 tab 1?
- localStorage seed 机制完全复用 (`baokuan_seed_text` 已实现)
- 爆款改写页 mount 时检测 seed → 自动填 textarea (现有逻辑)
- 唯一新增: tab 1 主按钮"提取并洗" 的语义需要爆款改写页**自动跳过 input step + 自动 analyze**
  - 类似 hotrewrite 已有的 `seedConsumedRef` 模式 (factory-hotrewrite-v2.jsx 行 64-90)
  - factory-baokuan-v2.jsx 当前是 mount 检测 seed + 填 textarea, **没有自动 analyze**
  - 需要补一个 effect: 如果 seed 来源是 make tab 1 提取路径, 自动调 doAnalyze

### 5.5 tab 2 文案太短/太长?
- 太短 (< 50 字): 提示"录音改写适合 200-1500 字, 太短 skill 抓不到结构"
- 太长 (> 2000 字): 提示"建议精简, 或拆成几段分别改"
- 不强拦, 让用户自己判断

## 九、风险 + 不做的事

### 已识别风险
1. **tab 切换的 wf snap 兼容**: D-016 有跨页持久化, 加 `activeTab` 字段需要确认 snap restore 不会把它覆盖回老值

3. **侧栏 3 个 skill 入口与 tab 是否冗余**: 短期保留 (power user); 长期看用户实际使用率, 数据决定要不要砍

### 明确不做
- ❌ 录音文件上传 (单独 backlog)
- ❌ 后端改动 (此次纯前端)
- ❌ 侧栏精简 (短期保留 power user 入口)
